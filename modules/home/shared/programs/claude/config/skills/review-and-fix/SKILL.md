# Review-and-Fix Skill

`/review` 결과를 기반으로 이슈를 하나씩 안내하며 수정하는 인터랙티브 워크플로우다.
사용자가 각 이슈를 직접 확인하고 수정 여부를 결정한다.

---

## 전제 조건

이 스킬을 실행하기 전에 `/emacs-dev` 스킬이 이미 실행되어 있어야 한다.
`/emacs-dev`가 ToolSearch로 `mcp__emacs-tools__*` 도구를 일괄 로드하므로,
`goto-line`도 이미 사용 가능한 상태다.

---

## 워크플로우

### 1단계: 이슈 목록 확인

현재 대화에 `/review` 결과가 있으면 그것을 사용한다.
없으면 `/review` 스킬을 먼저 실행한다.

이슈를 severity 순으로 정렬한다: `CRITICAL → HIGH → MEDIUM → LOW`

### 2단계: 이슈 큐 구성

이슈 목록을 내부적으로 큐로 관리한다. 각 이슈에 대해 3단계를 반복한다.

### 3단계: 이슈 안내 루프 (사용자 확인마다 한 번씩)

각 이슈에 대해 **반드시 아래 순서**로 진행한다:

#### 3-1. Emacs 네비게이션 (Emacs 모드일 때)

`goto-line` 도구를 호출한다:

```
file_path: /절대/경로/파일.go
line: 문제 코드의 첫 번째 라인 (1-based)
```

이 도구는 claude-code/magit 창을 피하고 코드 창을 찾아 파일을 열고
대상 라인을 창 상단 근처(5번째 줄)에 배치한다.

#### 3-2. 이슈 설명

아래 형식으로 상세히 설명한다:

```
## [SEVERITY] ID: 제목
📍 파일: `경로/파일.go:라인`

### 문제
무엇이 잘못됐는지 한 문장으로.

### 실행 경로
버그가 어떻게 발생하는지 코드 라인을 인용하며 단계별로 설명.
예: "1. A()가 호출되면 → 2. B 채널에 쓰기를 시도 → 3. 수신자가 없으면 영원히 블로킹"

### 수정 방향
변경할 내용과 이유.

### diff 미리보기
\`\`\`diff
- 기존 코드
+ 수정 코드
\`\`\`

진행: **'응 진행해줘'** / 스킵: **'스킵'**
```

#### 3-3. 사용자 응답 대기

사용자가 응답할 때까지 다음 작업을 하지 않는다.

- **'응 진행해줘'** (또는 긍정 응답): 수정 적용
- **'스킵'** (또는 부정/논의 응답): 이유를 메모하고 다음 이슈로

#### 3-4. 수정 적용

`Edit` 도구로 코드를 수정한다.

#### 3-5. 빌드 및 테스트 검증

언어에 맞는 빌드/타입체크 및 테스트 명령을 순서대로 실행한다:
- Go: `go build ./...` → `go test ./...`
- TypeScript: `tsc --noEmit` → `npm test` (또는 프로젝트의 테스트 명령)
- Python: `python -m py_compile` → `pytest`

빌드 또는 테스트 실패 시 즉시 원인을 분석하고 수정한다. 모두 통과 후 다음 이슈로.

### 4단계: 논리적 커밋 그룹화 (Emacs 모드)

관련된 이슈 수정이 쌓이면 (또는 사용자가 커밋을 요청하면) 아래 순서로 진행한다.

1. 파일 스테이징 — `git-stage`를 파일마다 호출:
   - `file_path`: 스테이징할 파일의 경로

2. 커밋 버퍼 준비 — `git-prepare-commit` 호출:
   - `message`: 커밋 메시지 (`fix(패키지): 변경 내용 요약` 형식)
   - 이 도구는 커밋 버퍼를 열고 메시지를 채우기만 한다. 커밋을 완료하지 않는다.

3. **여기서 멈추고 사용자에게 알린다:**
   > "커밋 메시지를 확인하고 `C-c C-c`로 커밋해주세요. 완료 후 다음 진행을 알려주세요."

   사용자가 직접 커밋을 완료하고 다음 진행 신호를 보낼 때까지 대기한다.

## 이슈 설명 품질 기준

- **실행 경로 필수**: "버그가 있다"가 아니라 "A → B → C 순서로 실행되면 D 조건에서 E가 발생한다"
- **코드 인용 포함**: 설명에 관련 코드 라인 번호와 스니펫을 명시
- **영향 범위 명시**: 이 버그가 언제, 어떤 조건에서 실제로 발생하는지
- **수정 후 동작 설명**: 수정하면 왜 문제가 해결되는지
- 이전 세션에서 사용자가 "설명이 부족하다"고 지적한 경우가 있었음 — 항상 충분한 맥락을 제공할 것

---

## 스킵/논의 처리

사용자가 이슈를 스킵하거나 "이건 의도된 동작이야"라고 말하면:

- 단순 스킵 요청: 이유를 메모하고 다음 이슈로 넘어간다.
- "의도된 동작"이라고 하더라도 **논리적 결함이 존재하는 경우** (예: goroutine 누수, 데드락 가능성, 리소스 미해제 등):
  1. 한 번 더 확인한다: 결함의 구체적인 발생 조건과 영향을 다시 설명하고 정말 의도한 것인지 묻는다.
  2. 사용자가 재확인 후에도 유지하겠다고 하면: 해당 코드에 의도를 명시하는 주석을 추가한다.

     주석 형식:
     ```go
     // NOTE: <결함 내용> — <의도된 이유 또는 설계 결정>
     ```

  3. 주석 추가 후 다음 이슈로 넘어간다. 스킵된 이슈는 이후에 다시 언급하지 않는다.

---

## 모든 이슈 완료 후

남은 미커밋 파일이 있으면 커밋을 제안한다.
커밋이 완료되면 수정 요약 (완료/스킵 목록)을 보고한다.
